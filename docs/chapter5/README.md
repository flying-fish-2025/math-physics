# 第五章 生态扩展与后续路线

当前仓库已经把 OpenClaw 的最小闭环跑通了：本地网关、模型 provider、飞书接入和基础排错。接下来该扩展什么，取决于你的目标，而不是“把所有开关都打开”。

## 1. skills 什么时候再开

建议在这两种情况下再配置 skills：

- 你已经有明确任务，需要调用额外依赖或外部能力
- 你已经知道哪些能力值得长期复用，而不是一次性尝试

如果你只是想先稳定聊天，skills 完全可以后置。

## 2. hooks 是做什么的

Hooks 本质上是自动化触发器，适合在命令执行前后做固定动作，例如：

- 新建会话时写入初始化上下文
- 重置时清理缓存
- 某类命令结束后把结果写入指定位置

建议在你已经熟悉 Agent 的工作流之后再开启。否则 hooks 只会让排错更复杂。

## 3. web search 什么时候配

只有当你的 Agent 真的需要联网检索，而且你也准备好了对应 API key 时，再去开启 `web_search`。否则它只是一个无效按钮。

## 4. 何时考虑后台服务

如果你满足下面两点，可以考虑把网关做成后台服务或开机启动：

- 你已经确认当前配置稳定
- 你确实需要长期在线，而不是临时调试

在此之前，继续使用前台运行的 `openclaw gateway run` 更利于排错。

## 5. 建议的扩展顺序

更合理的演进路径通常是：

1. 先稳定基础链路：网关、模型、渠道
2. 再增加可选能力：web search、skills
3. 然后增加自动化：hooks、定时任务
4. 最后再做多 Agent、多渠道和更复杂的工作流

## 6. 这一章的核心结论

OpenClaw 的问题从来都不是“能不能开更多功能”，而是“当前这套配置是否真的稳定、必要、可维护”。先把最小可用系统跑稳，再做扩展，这样效率最高。
