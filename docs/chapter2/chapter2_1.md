# 第二章 核心原理与系统架构

理解 OpenClaw，关键不是背命令，而是先看清它的几层结构。

## 1. OpenClaw 的核心组成

一个最小可用的 OpenClaw 系统，通常包含五部分：

- `Gateway`：本地运行的网关进程，负责承接 Web UI、消息通道和 Agent 调度
- `Agent`：真正执行任务的智能体，包含模型、上下文、工具和工作区
- `Channel`：对接飞书、Slack、Discord 等消息渠道
- `Model Provider`：提供模型推理能力的 API 或本地模型服务
- `Workspace`：Agent 的本地工作目录，用来保存文件、会话和上下文

其中最容易被误解的是 `Gateway`。很多用户以为 `openclaw onboard` 跑完后系统就会一直在线，实际上不是。`onboard` 更接近初始化向导，真正让系统保持可用的是 `openclaw gateway run` 或后台服务方式。

## 2. 典型工作流

以“飞书里 @机器人 问一个问题”为例，大致流程如下：

1. 飞书把消息通过长连接或事件通道送到 OpenClaw
2. `Gateway` 接收消息并交给对应 `Channel`
3. `Channel` 把消息路由到目标 `Agent`
4. `Agent` 调用配置好的模型 provider 生成回复
5. 如果需要，Agent 再调用工具或技能补充执行
6. 最终结果再经由 `Gateway` 回发到飞书

只要其中任何一层出问题，表面上看到的现象就可能都是“机器人没回消息”。因此排错时不要只盯着聊天窗口，而要按链路逐层看。

## 3. 为什么会出现“装好了但不能用”

这是 OpenClaw 新手最常见的问题，原因通常不是单点，而是链路中某一层没配置完整。

典型例子：

- 命令行找不到 `openclaw`：安装目录不在 `Path`
- Dashboard 打不开：网关进程没在运行
- Dashboard 能打开但发消息报错：模型 provider 不可用
- 飞书私聊不回：pairing 没完成
- 飞书收到消息但后续动作失败：应用权限 scope 不够

因此，正确思路不是“重装”，而是先定位问题落在哪一层。

## 4. Skill、Hook、Tool 分别是什么

### 4.1 Tool

`Tool` 是能力接口，模型可以调用它完成外部动作，例如搜索、读写文件、访问第三方 API。

### 4.2 Skill

`Skill` 更像预封装好的能力包，通常包含依赖、参数配置和一组可复用的功能模块。是否启用 skill，取决于你是否真的需要这类能力。

### 4.3 Hook

`Hook` 是命令级自动化触发器。它不是必须组件，适合在你已经清楚自己的工作流之后再用。比如：

- 执行 `/new` 时自动保存上下文
- 会话重置时清理临时状态
- 某类任务结束后自动归档

对于刚入门的使用者，建议第一轮全部跳过。先把基础消息收发跑通，再考虑自动化。

## 5. 为什么说 OpenClaw 更像平台而不是聊天机器人

因为它的关注点不是“回答得像不像人”，而是：

- 如何对接外部渠道
- 如何管理 Agent 的状态
- 如何安全地开放工具权限
- 如何让模型调用结构化能力
- 如何把一个系统稳定地跑起来

这也是为什么你一旦把 OpenClaw 跑顺了，它就不只是一个机器人，而可以作为自己的本地 Agent 基础设施。
